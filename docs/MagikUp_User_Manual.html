1. Introduzione

1.1 Panoramica

Magiku(r)p è un’applicazione web per la gestione di backup, restore e trasferimento di database PostgreSQL. Supporta connessioni dirette e connessioni tramite tunnel AWS SSM attraverso jump host EC2 configurabili.

Funzionalità principali:

  • Backup — Backup completi o per schema con output in tempo reale
  • Restore — Ripristino con selezione schema e remapping ruoli
  • Transfer — Copia database one-click tra qualsiasi endpoint
  • Query Editor — Esecuzione query SQL con object browser e esportazione risultati
  • File Manager — Upload, download e gestione file di backup (fino a 5 GB)
  • Multi-Utente — Tre ruoli (Admin, Operator, Viewer) con permessi granulari
  • Sicurezza — Rate limiting, lockout account, policy password, audit log
  • SSM Tunnel — Port forwarding attraverso EC2 jump host con AWS Systems Manager

1.2 Requisiti

Per il deploy Kubernetes

  • Cluster Kubernetes con kubectl configurato
  • Nginx Ingress Controller installato
  • Storage class disponibile per i Persistent Volume Claim
  • Accesso a un container registry (es. GHCR)

Per l’uso

  • Browser moderno (Chrome, Firefox, Edge, Safari)
  • Credenziali di accesso fornite dall’amministratore
  • (Opzionale) Credenziali AWS con permessi SSM per tunnel

1.3 Architettura

L’applicazione è composta da:

  • Backend: FastAPI (Python) con WebSocket per streaming real-time
  • Frontend: HTML/CSS/JS con Bootstrap 5 e tema chiaro/scuro
  • Storage: File system per backup e configurazione (PVC in Kubernetes)
  • Autenticazione: Sessioni firmate con token, gestione utenti su file JSON
Browser (UI)
    │ HTTP / WebSocket
    ▼
FastAPI (main.py) ── Auth (auth.py) ── Users (users.json)

    ├── DB Service (db_service.py) ── psycopg2
    └── AWS Service (aws_service.py) ── boto3

2. Installazione e Deploy

2.1 Deploy su Kubernetes

Build e Push dell’immagine

# Build dell'immagine Docker
docker build -t ghcr.io/fpellizz/magikup:3.4.0 .

# Push al registry
docker push ghcr.io/fpellizz/magikup:3.4.0

Configurazione Secrets

Copiare il template del secret:
cp kubernetes/secret.yaml.example kubernetes/secret.yaml
Generare la chiave di cifratura Fernet:
python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"
Sostituire REPLACE_WITH_GENERATED_KEY in secret.yaml con la chiave generata.

Configurazione Ingress

Modificare il file kubernetes/ingress.yaml e sostituire magikup.example.com con l’hostname reale.

Context Path: Per il deploy sotto un prefisso URL (es. /magikup), impostare la variabile d’ambiente ROOT_PATH nel file kubernetes/deployment.yaml e configurare le regole di path nell’ingress. Vedi la sezione 2.5 Context Path / Reverse Proxy per i dettagli.

Deploy con Kustomize

# Applica tutti i manifest
kubectl apply -k kubernetes/

# Verifica lo stato
kubectl get pods -l app=magikup
kubectl logs deploy/magikup

Accesso all’applicazione

# Via port-forward (per test)
kubectl port-forward svc/magikup 8000:8000

# Via Ingress (produzione)
# Accedere a https://magikup.tuodominio.com

2.3 Sviluppo Locale

pip install -r requirements.txt
python run.py --reload

L’applicazione sarà disponibile su http://localhost:8000.

2.4 Primo Avvio

Al primo avvio l’applicazione:

  1. Crea automaticamente il file users.json con l’utente admin predefinito
  2. Genera una password casuale e la stampa nei log dell’applicazione (cercare la riga INITIAL ADMIN PASSWORD)
Come trovare la password iniziale: Consultare i log all’avvio dell’applicazione. La password viene stampata tra una serie di ==== con il messaggio INITIAL ADMIN PASSWORD: <password>.
Con Docker: docker logs <container_name> 2>&1 | grep "INITIAL ADMIN"
Importante! Cambiare immediatamente la password dell’utente admin dopo il primo login.

2.5 Context Path / Reverse Proxy

MagikUp supporta il deploy sotto un prefisso URL (context path) per configurazioni con reverse proxy. Ad esempio, l’applicazione può essere accessibile all’indirizzo https://example.com/magikup/ anziché alla root del dominio.

Configurazione

Il context path può essere configurato in tre modi (in ordine di priorità):

  1. Variabile d’ambiente ROOT_PATH (priorità massima) — Impostare nel Docker Compose, nel deployment Kubernetes o nella shell. Questo è l’approccio raccomandato per i deploy in container.
  2. Parametro context_path in config.ini nella sezione [settings] — Configurabile dal pannello Admin (tab Settings). Richiede il riavvio dell’applicazione per avere effetto.
  3. Default: vuoto — L’applicazione viene servita al path root /.

Esempio Docker Compose

services:
  magikup:
    image: magikup:latest
    environment:
      - ROOT_PATH=/magikup
    ports:
      - "8000:8000"

Esempio Nginx Reverse Proxy

location /magikup/ {
    proxy_pass http://localhost:8000/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    # WebSocket support
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}

Esempio Kubernetes Ingress

Per il routing basato su path nell’ingress, decommentare le annotazioni e le regole di path rilevanti in kubernetes/ingress.yaml e impostare ROOT_PATH in kubernetes/deployment.yaml:

# In deployment.yaml, aggiungere alla sezione env:
- name: ROOT_PATH
  value: "/magikup"

Configurazione da Admin UI

Il context path può essere visualizzato e configurato dal pannello Admin in Settings > Network.

Nota: Se ROOT_PATH è impostato come variabile d’ambiente, ha la precedenza sul valore in config.ini. Nella UI questo viene indicato con un badge informativo. Le modifiche al context path richiedono un riavvio dell’applicazione per avere effetto.

Note Importanti

  • L’endpoint di health check /health funziona sempre a livello di container senza il prefisso.
  • Quando si utilizza un reverse proxy, il proxy deve rimuovere il prefisso del context path prima di inoltrare le richieste all’applicazione.
  • Anche le connessioni WebSocket sono context-path aware.

3.5 Sicurezza: Rate Limiting e Lockout

Rate Limiting (per IP)

Dopo 5 tentativi di login falliti dallo stesso indirizzo IP, l’accesso viene bloccato per 5 minuti. Viene mostrato un messaggio che indica il tempo rimanente.

Account Lockout (per utente)

Dopo 10 tentativi di login falliti per lo stesso username, l’account viene bloccato. Solo un amministratore può sbloccarlo dal pannello Admin → Users.

Audit Log

Tutti gli eventi di sicurezza vengono registrati:

  • Login riusciti e falliti
  • Cambio password
  • Creazione/modifica/eliminazione utenti
  • Blocco e sblocco account

Parallel Jobs

Il parametro --jobs N permette di eseguire il restore usando N processi paralleli (da 1 a 8). Questo può accelerare significativamente il ripristino di database di grandi dimensioni su server multi-core.

Nota: L’uso di --jobs è incompatibile con --single-transaction.

Esclusioni

Due aree di testo nella sezione Exclusions permettono di escludere schema e tabelle dal restore:

CampoDescrizione
Exclude schemasUno schema per riga. Ogni valore viene passato come --exclude-schema a pg_restore.
Exclude tablesUn pattern per riga. Supporta wildcard * (qualsiasi carattere) e ? (singolo carattere).

L’esclusione tabelle utilizza il filtraggio TOC (Table of Contents): pg_restore esegue --list per ottenere l’elenco degli oggetti nel backup, le voci corrispondenti ai pattern vengono commentate, e il file filtrato viene passato con --use-list. Questo approccio è necessario perché pg_restore non dispone di un flag nativo --exclude-table.

Suggerimento: Le opzioni --no-owner, --no-privileges e --no-tablespaces sono consigliate quando si ripristina su un server diverso dall’originale, dove utenti, ruoli e tablespace potrebbero non esistere.

6.4 Role Mapping

Il campo “Role” permette di assegnare l’ownership di tutti gli oggetti ripristinati a un ruolo specifico del database di destinazione. Il menu a tendina elenca i ruoli disponibili sull’endpoint selezionato, evidenziando i superuser.

Se non si seleziona un ruolo, pg_restore userà l’utente di connessione dell’endpoint.

6.5 Restore TimescaleDB

Nella sezione Advanced Parameters è disponibile il toggle TimescaleDB. Quando attivo, il restore viene adattato alle specifiche di un database con estensione timescaledb installata, seguendo la procedura ufficiale TimescaleDB.

Flusso attivato dal toggle:
  1. L’applicazione si connette al database di destinazione ed esegue SELECT timescaledb_pre_restore(); — disabilita temporaneamente i background worker e rimuove le restrizioni sugli hypertable.
  2. Viene eseguito pg_restore aggiungendo il flag --disable-triggers (utile per evitare l’esecuzione di trigger di integrità referenziale durante il caricamento dati).
  3. Al termine del restore, viene eseguito SELECT timescaledb_post_restore(); — ripristina le protezioni e riattiva i background worker.

Prerequisiti

  • Il database di destinazione deve già avere l’estensione timescaledb installata (CREATE EXTENSION IF NOT EXISTS timescaledb;).
  • L’utente di connessione deve avere privilegi di superuser (richiesto sia da timescaledb_pre_restore() sia da --disable-triggers).
  • La versione di TimescaleDB sul target deve essere uguale o superiore a quella sorgente.

Gestione errori

ErroreComportamento
timescaledb_pre_restore() fallisce
(estensione mancante, utente non superuser)		 Il restore viene abortito prima di eseguire pg_restore. Lo stato finale è failed con il messaggio d’errore originale.
pg_restore fallisce Errore riportato normalmente. timescaledb_post_restore() viene comunque tentata per riattivare i worker.
timescaledb_post_restore() fallisce Lo stato dell’operazione viene marcato come failed anche se pg_restore è terminato correttamente, perché il database resta in stato inconsistente (background worker non riattivati).

Il toggle TimescaleDB è disponibile anche nella sezione Advanced Parameters della fase Restore della pagina Transfer, consentendo di attivare lo stesso flusso in operazioni di trasferimento completo.

11. SSM Tunnel

11.1 Come Funziona

AWS Systems Manager (SSM) permette di creare tunnel di port forwarding attraverso istanze EC2 senza bisogno di accesso SSH diretto. Questo consente di raggiungere database RDS/Aurora in reti private. 

MagikUp Pod ──> SSM Session ──> EC2 Jump Host ──> VPC Network ──> RDS/Aurora
(localhost:15432)                 (SSM Agent)                    (db.rds:5432)

Flusso operativo

  1. L’utente avvia un’operazione su un endpoint con SSM attivo
  2. L’applicazione apre automaticamente un tunnel SSM attraverso il jump host configurato
  3. pg_dump/pg_restore si connettono al database attraverso il tunnel
  4. Se esiste già un tunnel attivo per lo stesso endpoint, viene riutilizzato
  5. Il tunnel rimane attivo fino alla chiusura manuale o al riavvio del pod

11.2 Configurazione

Prerequisiti

  • Credenziali AWS configurate nel tab AWS Accounts con permessi ssm:StartSession
  • Istanza EC2 con SSM Agent attivo e in stato “Online”
  • Rete: il jump host deve poter raggiungere il database target (security group, routing)

Passi di configurazione

In Admin → AWS Accounts: aggiungere le credenziali AWS e verificare con “Test Connection”.
In Admin → Jump Hosts: aggiungere il jump host EC2 con Instance ID e account AWS associato.
In Admin → Endpoints: creare l’endpoint con “Use SSM Tunnel” attivo e selezionare il jump host.
Testare la connessione con il pulsante “Test Connection” sull’endpoint.
Suggerimento: Nella dashboard, gli endpoint con SSM attivo mostrano un badge blu “SSM”. Nella pagina Backup/Restore/Transfer, un banner informativo mostra il jump host in uso.
